対象DB: SQLite（React Native / Expo）
対応SQLファイル: `chill_night_town_スキーマ_v2.sql`

## 5. study_tag（学習内容タグ）

**分類: マスタ（ユーザー追加可）**

学習内容タグを管理するテーブル。標準タグとユーザーが作成したマイタグを保持する。

| カラム名      | データ型     | PK  | FK      | NULL | 初期値          | 説明                                                 |
| ------------- | ------------ | :-: | ------- | :--: | --------------- | ---------------------------------------------------- |
| id            | INTEGER      | ○  |         | 不可 | 自動採番        | タグID（主キー）                                     |
| user_id       | INTEGER      |     | user.id | 可   | なし            | マイタグの所有者（NULL=標準タグ）。ON DELETE CASCADE |
| name          | TEXT         |     |         | 不可 | なし            | タグ名                                               |
| is_custom     | INTEGER(0/1) |     |         | 不可 | 0               | 1=マイタグ（自由入力）、0=標準タグ                   |
| is_active     | INTEGER(0/1) |     |         | 不可 | 1               | 有効/無効フラグ（マイタグの論理削除用）              |
| display_order | INTEGER      |     |         | 不可 | 0               | 表示順                                               |
| created_at    | TEXT         |     |         | 不可 | datetime('now') | 作成日時                                             |

**制約補足**: 標準タグは`name`が全体で一意、マイタグは`(user_id, name)`単位で一意。マイタグは1ユーザー最大20件（`trg_study_tag_mytag_limit`トリガーで制御。復活時は`trg_study_tag_mytag_limit_revive`で同様に担保）。

**仕様補足（v2ルールとの対応）**:
- 標準タグの投入データは6件: 資格勉強／レポート・課題／暗記・復習／プログラミング／読書／その他（要件3.4）。
- 重複判定（登録・編集時）は標準タグ・マイタグの両方を対象とする（アプリ側検証＋UNIQUE制約）。
- 論理削除済み（`is_active=0`）のマイタグと同名が入力された場合は、`is_active=1`へ更新して復活させる（新規行は作らない）。
- **上限20件のカウントは`is_active=1`のマイタグのみを対象とする**（削除済みは上限に含めない）。
- 名称変更は`name`の更新のみで過去記録の表示に反映される（記録側はIDで参照しているため）。

## 9. growth_level_threshold（レベルアップ閾値マスタ）

街のレベルアップ条件を管理するマスタテーブル。成長方式ごとに各レベルへ到達するために必要な条件を保持する。他テーブルとの外部キー関係を持たない独立マスタ。

| カラム名       | データ型 | PK  | FK  | NULL | 初期値   | 説明                                 |
| -------------- | -------- | :-: | --- | :--: | -------- | ------------------------------------ |
| id             | INTEGER  | ○  |     | 不可 | 自動採番 | 閾値ID（主キー）                     |
| method         | TEXT     |     |     | 不可 | なし     | 成長方式。CHECK: `habit`/`project`   |
| level          | INTEGER  |     |     | 不可 | なし     | 到達レベル。CHECK: 2〜5              |
| required_value | INTEGER  |     |     | 不可 | なし     | 到達に必要な値。CHECK: 0より大きい値 |

**制約補足**: `UNIQUE(method, level)`。

**運用上の補足**:
- `habit`（習慣型）: 本テーブルの`required_value`（必要累計経験値）を使用する。レベルアップに必要な経験値は一律5のため、投入データは Lv.2=5 / Lv.3=10 / Lv.4=15 / Lv.5=20（累計値）。バランス調整はこのマスタの更新のみで行える（要件6.2①）。
- `project`（プロジェクト型）: レベル基準は`town_progress.project_target_minutes`から動的算出するため、本テーブルの`project`行は現段階では使用しない（データ投入不要）。将来的な仕様変更・拡張性を考慮し、テーブル構造は維持する。

**備考**: 現段階では5段階まで実装する。将来的には10段階以上へ拡張可能な設計とする。

## 15a. daily_night_weather（学習日ごとに選択された夜の天気）【新規】

**その学習日の「今夜の天気」を、学習セッションとは独立して保持するテーブル。** ホーム画面の背景演出・環境音は本テーブルを参照する（要件8章）。

**仕様補足**:

- **本テーブルを新設した理由**: 旧版では演出用の天気を「当学習日の `study_session` / `active_session` の `night_weather_id`」から導出していたため、**学習を開始しない限り天気を選べなかった**。ホーム画面から天気のみを選択できるようにするにあたり（要件2.5）、セッションから独立した保存先が必要になった。

- 天気が選択・変更される経路は次の3つ。いずれも本テーブルを更新する:
  1. ホーム画面の天気カード（要件2.5）
  2. タイマー設定モーダル（要件3.1）— セッション開始時に `active_session` へも複写する
  3. 学習成果記録画面（要件3.4）— 当該セッションの `night_weather_id` も更新する

## 補足事項

- 真偽値カラムはすべて`INTEGER`型（0=false／1=true）で表現している（SQLiteに`BOOLEAN`型が存在しないため）。
- 日時カラムはすべて`TEXT`型（ISO8601形式）、日付のみのカラムは`TEXT`型（`YYYY-MM-DD`形式）で統一している。
- 「学習日」カラム（`study_session.study_date`／`additional_study_time.target_date`／`daily_goal_achievement.study_date`）はすべて要件0章の定義（18:00〜翌5:00サイクルの開始日）に従う。算出はアプリ側で共通関数化することを推奨する。
- `user`テーブルの行を削除すると、`ON DELETE CASCADE`により関連するユーザー生成データ（学習記録・進捗・各種設定・マイタグ・延長宣言・計測状態・達成記録・選択済みの夜の天気等）が連動削除される（10.8 データ削除機能に対応）。マスタデータ（town / night_weather / emotion / npc / npc_message / ambient_sound / growth_level_threshold）は削除されない。
- 街の成長段階は現段階（MVP）では5段階（Lv1〜Lv5）まで実装する。将来的には10段階以上へ拡張可能な設計とする（`growth_level_threshold.level`および`town_progress.current_level`のCHECK範囲拡張で対応予定）。
- ニックネーム・サブタイトルの文字数上限（各20文字）はアプリ側の入力検証で担保する。
