# CLAUDE.md — Chill Night Town プロジェクト前提

このリポジトリは、夜間学習者向けの感情支援型学習アプリ「Chill Night Town」（卒業研究・チーム開発）である。
コンセプトは「勉強を管理する」ではなく「またこの世界に帰ってきたい夜の街」。静かで落ち着いた世界観を最優先し、ユーザーを急かす・責める・競わせる実装は行わない。

## 技術スタック

- Expo（React Native）+ Expo Router
- TypeScript（strict設定）
- SQLite（expo-sqlite）— サーバーなしの完全ローカル構成
- 状態管理: Context API 3本構成（`contexts/TimerContext.tsx` / `contexts/AudioContext.tsx` / `contexts/SettingsContext.tsx`）。外部の状態管理ライブラリは使わない
- 使用パッケージ: expo-sqlite / expo-notifications（ローカル通知のみ）/ expo-audio / expo-battery / expo-keep-awake / expo-screen-orientation

## 重要な用語定義（頻出・間違えやすい）

- **学習日** = 18:00〜翌5:00の1サイクル。セッションの**開始時刻**が属するサイクルの開始日に帰属する（例: 1/10 23:30開始→翌1:30終了は「1/10の記録」）。カレンダー・目標達成判定・経験値付与はすべて学習日単位。算出は共通関数に集約する
- **実績学習時間** = 一時停止中・ポモドーロの休憩フェーズを除いた作業時間の合計。成長・判定はすべてこの値を使う
- **タイマー計測は時刻差分方式**（カウンター変数は使わない）。開始・一時停止・再開の時刻を都度 `active_session` に永続化し、経過時間は「現在時刻 − 開始時刻 − 一時停止累積」で算出。差分が負なら0に丸める
- **実績1分未満のセッションは保存せず破棄**（終了演出・成果記録に進まない）
- **経験値は1学習日につき最大1回・付与後は取り消さない**（`daily_goal_achievement` に付与済みを記録）
- **街のレベルは一度上がったら下がらない**（過去最高レベルを維持）
- 5:00到達でタイマーは自動終了。実績は5:00まで
- 成長方式は**習慣型**（目標達成日に経験値+1）と**プロジェクト型**（街ごとの目標時間へ累計加算）。方式はアプリ全体で共通、実績（累計時間・経験値）は街ごとに個別管理し、累計時間は方式に関わらず常に加算する
- **タイマー稼働中は、判定・記録に影響する設定を変更不可**（一日の目標時間・成長方式・プロジェクト型目標時間・街の切替・データ初期化）。音量・ニックネーム等は変更可

## コーディング規約

- Prettier準拠（保存時自動整形）。ESLintの警告は放置しない
- TypeScriptの `any` は原則使わない。DBの行データにも型を定義する
- コメント・コミットメッセージは日本語でよい
- パッケージ追加は `npx expo install` を使う（npm install直はSDK非互換の恐れ）
- DBスキーマの変更が必要な場合は、**先にテーブル定義書とスキーマSQLを更新してから**コードを変更する（ドキュメントが正）
- マジックナンバーを避ける。値域・既定値（10〜720分、経験値5、3〜27人など要件定義書記載の数値）は定数として一元管理する
- 1つのContextに機能を詰め込みすぎない（Timer / Audio / Settings の責務分離を守る）
- UI文言はコンセプト準拠：責めない・急かさない・感嘆符を多用しない

## テスト

`npm test`（jest-expo）。対象は **`src/lib` 配下の純粋なロジック**に絞る。

- **テストを書く対象**: 学習日の帰属（`study-day.ts`）、タイマー計測（`timer.ts`）、入力検証（`validation.ts`）。いずれも**画面を見ても正しさが分からず、壊れると記録・成長判定に直接響く**箇所
- **テストを書かない対象**: 画面（UI）・DBアクセス・Context。これらは実機での目視確認とする（MVPの工数配分としてUIテストは持たない）
- 値域・境界（10分ちょうど／721分／4:59と5:00／59秒と60秒 等）は**ドキュメントが正**。テストはドキュメントとのズレを検出するために書く
- ロジックを変更したら、**先にテストを更新してから**実装を変える（仕様変更の場合はドキュメント → テスト → 実装の順）
- 検証は使い捨てスクリプトで済ませず、**再実行できる形でリポジトリに残す**（卒業研究として検証手段を示せるようにするため）

## 作業上の注意

- **仕様に定義のない挙動を実装する必要が生じた場合は、勝手に決めずに人間へ確認を求める**（本プロジェクトは未定義事項を潰し切ってあるため、「仕様に書いていない」は大抵ドキュメントの読み落としか、本当に要検討の新規事項のどちらか）
- コミット前に `npx tsc --noEmit` / `npm run lint` / `npm test` を通す。UIを伴う変更は、加えて**実機確認でOKが出てからコミットする**
- 大きめの変更はPlanモードで計画を提示してから実装する
- 生成したコードの採否は人間がレビューして決める（卒業研究として説明責任があるため、動くだけのブラックボックスを作らない）
